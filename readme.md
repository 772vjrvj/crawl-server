# crawl-server 구축 정리

## 1. 작업 목적

외부 고객 프로그램이 내 PC에서 실행 중인 Python API 서버를 호출할 수 있도록 `crawl-server`를 구축했다.

즉 목표는 다음과 같다.

- 내 PC에서 Python API 서버 실행
- 외부 인터넷에서 해당 서버로 접근 가능하게 구성
- 최소한의 인증(API Key) 적용
- 이후 실제 업무 로직(Bizno 등)을 이 서버에 붙일 수 있는 상태 만들기

현재는 네트워크와 서버 뼈대를 먼저 잡고,  
외부망에서 `/health`, `/api/ping` 테스트까지 성공한 상태다.

---

## 2. 왜 Flask로 시작했는가

처음에는 가장 빠르게 API 서버를 띄우기 위해 Flask를 사용했다.

### 이유
- Python에서 가장 빠르게 웹 API를 만들 수 있음
- 테스트용 라우트(`/health`, `/api/ping`)를 금방 붙일 수 있음
- 이후 Bizno 로직이나 Selenium 로직도 Python 안에서 자연스럽게 연결 가능

즉 Flask는 **빠르게 API 서버를 검증하기 위한 출발점**으로 사용했다.

---

## 3. 왜 Waitress를 사용했는가

처음에는 Flask 기본 실행 방식(`app.run`)으로 서버를 띄웠다.  
이 방식은 개발 단계에서는 편하지만, 외부 고객이 실제로 접근하는 서버로 쓰기에는 적절하지 않다.

Flask 실행 시 아래와 같은 경고가 나온다.

- development server
- production deployment 용으로 쓰지 말 것

### Waitress를 사용한 이유
Waitress는 Python WSGI 서버로, Flask 앱을 **운영용 형태로 실행**할 수 있게 해준다.

즉 역할은 다음과 같다.

- Flask 앱을 외부 요청을 받을 수 있는 서버로 안정적으로 실행
- 개발서버보다 운영 환경에 더 적합
- `0.0.0.0:5000` 으로 외부 요청 대기 가능
- 구조를 `main.py`(앱) / `run_server.py`(실행) 형태로 분리 가능

정리하면:

- `Flask` = API를 만드는 프레임워크
- `Waitress` = 그 Flask 앱을 운영용으로 실행하는 서버

그래서 최종적으로는 Flask 개발서버 대신 Waitress를 사용했다.

## 4. 왜 Windows 방화벽을 열었는가

서버가 내 PC에서 떠 있어도, Windows 방화벽이 해당 포트를 막고 있으면  
다른 PC나 외부 인터넷에서는 접근할 수 없다.

### 방화벽이란
방화벽은 내 PC로 들어오는 네트워크 연결을 제한하는 보안 장치다.

쉽게 말하면:

- 내 PC가 5000 포트에서 서버를 실행 중이어도
- Windows 방화벽이 “외부에서 이 포트로 들어오는 연결”을 막으면
- 서버는 살아 있어도 외부에서는 접속 실패가 난다

### 왜 5000 포트를 허용했는가
이번 서버는 `5000` 포트에서 실행했기 때문에,  
해당 포트로 들어오는 TCP 연결을 허용해야 했다.

실행한 명령은 다음과 같다.

```bash
netsh advfirewall firewall add rule name="crawl-server-5000" dir=in action=allow protocol=TCP localport=5000
```
이 작업의 의미

이 설정은 다음 의미를 가진다.

내 PC의 5000 포트로 들어오는 요청 허용

서버 프로그램이 외부 요청을 받을 수 있도록 허용

최소한 PC 내부 보안 차단을 해제

즉, 방화벽 오픈은 “내 PC가 외부 요청을 받을 수 있게 만드는 첫 번째 단계” 였다.

5. 왜 내부 IP와 게이트웨이를 확인했는가

외부에서 내 서버에 접근시키려면 먼저
“내 서버가 집/사무실 네트워크 안에서 어느 주소를 쓰는지” 알아야 한다.

ipconfig 결과로 확인한 값은 다음과 같다.

서버 PC 내부 IP: 172.30.1.8

기본 게이트웨이: 172.30.1.254

이 값들이 중요한 이유
내부 IP (172.30.1.8)

이건 내 API 서버가 실제로 떠 있는 내 PC의 로컬 네트워크 주소다.

즉 공유기는 외부 요청을 결국 이 주소로 넘겨야 한다.

기본 게이트웨이 (172.30.1.254)

이건 보통 공유기 주소다.

즉 공유기 관리자 페이지에 접속할 때 이 주소를 사용한다.

정리

내 서버 주소: 172.30.1.8:5000

공유기 주소: 172.30.1.254

이 정보를 알아야 포트포워딩 설정을 정확히 할 수 있다.

6. 왜 공유기 관리자 페이지(KT 홈허브)에 들어갔는가

내 PC에서 서버를 띄운 것만으로는 외부 인터넷에서 바로 접속되지 않는다.

왜냐하면 외부에서 오는 요청은 먼저 공유기에 도착하기 때문이다.

즉 구조는 대충 이렇다.

외부 고객 프로그램 → 공인 IP로 접속

요청은 먼저 공유기에 도착

공유기가 그 요청을 내 PC로 넘겨줘야 함

그런데 공유기는 기본적으로
“외부에서 들어온 요청을 어느 내부 PC로 보낼지” 모른다.

그래서 공유기 관리자 페이지에 접속해서
그 규칙을 직접 설정해야 했다.

접속한 이유

KT 홈허브 관리자 페이지에 들어간 이유는 다음과 같다.

현재 네트워크를 관리하는 장비 확인

공인 IP를 실제로 어떤 장비가 받고 있는지 확인

포트포워딩 설정 메뉴 찾기

외부 포트를 내 PC로 연결하는 규칙 만들기

즉, 공유기는 외부 인터넷과 내 내부망 사이의 관문이기 때문에 반드시 설정이 필요했다.

7. 왜 포트포워딩을 했는가

포트포워딩은 이번 작업의 핵심이다.

포트포워딩이란

공유기로 들어온 외부 요청을
내부 네트워크의 특정 PC와 특정 포트로 전달하는 기능이다.

쉽게 말하면:

외부에서 공인IP:5000 으로 접속하면

공유기가 그 요청을 받아서

내부의 172.30.1.8:5000 으로 넘겨주는 것

왜 필요했는가

내 API 서버는 내 PC(172.30.1.8)에 떠 있다.
하지만 외부 고객은 이 내부 주소를 직접 알 수도 없고, 접근도 못 한다.

외부에서는 오직 공인 IP 로만 접근 가능하다.

그래서 공유기에게 이렇게 알려줘야 한다.

외부에서 5000 포트로 들어오는 요청은
내부의 172.30.1.8:5000 으로 보내라

실제 등록한 값

외부 포트: 5000

내부 IP: 172.30.1.8

내부 포트: 5000

프로토콜: TCP

이 설정의 의미

결국 다음 연결을 만든 것이다.

외부 공인IP:5000

내부 172.30.1.8:5000

즉, 포트포워딩이 있어야 외부 인터넷 요청이 내 PC 서버까지 도착할 수 있다.

8. KT 모드 / 공유기 모드를 왜 확인했는가

공유기 설정을 보다 보니 KT 모드, 공유기 모드, 브릿지 모드 같은 항목이 있었다.

이걸 확인한 이유는 처음에 포트포워딩 메뉴가 보이지 않았기 때문이다.

왜 확인했는가

보통 장비 모드에 따라 다음이 달라질 수 있다.

NAT 기능 활성 여부

포트포워딩 메뉴 노출 여부

내부 IP 할당 방식

장비가 공유기 역할을 하는지 여부

처음에는 KT 모드 때문에 포트포워딩이 제한되는지 의심해서
공유기 모드 전환 가능 여부도 같이 확인했다.

최종 결론

최종적으로는 같은 관리자 페이지 안에서
포트포워드 설정 메뉴를 찾아 직접 규칙을 추가하면서 해결했다.

즉 핵심은:

모드를 확인한 이유는 “포트포워딩이 왜 안 보이는지” 찾기 위해서였고

실제 해결은 “포트포워드 메뉴를 찾아 규칙을 등록” 하면서 완료되었다

9. 왜 /health 와 /api/ping 을 테스트했는가

외부 서버 구축은 한 번에 다 붙이면 어디서 막히는지 알기 어렵다.
그래서 가장 단순한 API부터 단계별로 검증했다.

/health

이 API는 서버가 살아 있는지 확인하는 용도다.

확인 가능한 것:

서버 실행 여부

포트 연결 여부

외부 접근 여부

즉 /health 는 “서버 자체가 열려 있나?” 확인하는 최소 테스트다.

/api/ping

이 API는 실제 업무 API 전 단계 테스트다.

확인 가능한 것:

POST 요청 가능 여부

JSON body 전달 여부

API Key 인증 여부

외부 프로그램 호출 구조 정상 여부

즉 /api/ping 은 “실제 API 형태로 외부 요청이 잘 들어오나?” 를 확인하기 위한 테스트였다.

10. 왜 API Key를 넣었는가

외부에 API 서버를 열면 누구나 해당 주소를 두드릴 수 있다.
그래서 최소한의 인증 장치가 필요했다.

API Key를 넣은 이유

아무나 API를 호출하지 못하게 하기 위해

고객 프로그램만 올바른 키를 넣어서 호출하게 하기 위해

초기에 가장 간단하게 적용 가능한 인증 방식이기 때문에

현재는 헤더 X-API-KEY 로 인증하도록 구성했다.

즉, 외부 공개 상태라도 최소한 “키가 없으면 거절” 하는 구조를 먼저 만든 것이다.

11. 왜 공인 IP를 확인했는가

외부 고객은 내 집/사무실 내부 IP(172.30.1.8) 로는 접근할 수 없다.
외부에서는 오직 공인 IP 로만 접속할 수 있다.

그래서 현재 인터넷에서 내 네트워크가 어떤 주소로 보이는지 확인해야 했다.

공인 IP를 확인한 이유

외부 테스트 주소를 알아내기 위해

고객 프로그램이 접속할 주소를 파악하기 위해

포트포워딩 후 실제 외부 접근 테스트를 하기 위해

중간에 공인 IP가 바뀌는 것도 확인했다.
즉 지금 환경은 고정 IP가 아니라 유동 IP일 가능성이 높다.

이것은 나중에 다음 문제와 연결된다.

공인 IP가 바뀌면 기존 접속 주소가 바뀜

Cafe24 같은 외부 서비스에 허용 IP 등록해놨다면 접속이 끊길 수 있음

12. 외부 테스트 결과

최종적으로 외부망에서 다음 테스트를 성공했다.

/health

모바일 데이터 / 다른 네트워크에서 접속 성공

외부에서 서버 응답이 들어오는 것 확인

/api/ping

외부 POST 요청 성공

API Key 인증 성공

JSON body 수신 성공

이 결과로 다음이 모두 검증되었다.

Waitress 서버 정상

Windows 방화벽 설정 정상

공유기 포트포워딩 정상

외부 공인 IP 진입 정상

인증 정상

실제 JSON API 요청/응답 정상

13. 현재 완료 상태

현재까지 완료된 핵심은 다음과 같다.

/health 구현

보안 측
